# CAE ML Suite: Comprehensive Configuration Reference

> Live-code audit snapshot: 2026-07-19, from
> `C:\Users\Lee\Desktop\Huni\CAE_ML_Suite`.

This is the configuration source of truth for the current suite. It covers the
unified launcher, every model backend, all flat-text config keys, all checked-in
config templates, auxiliary Python CLIs, shell/environment controls, local TOML
settings, runtime-injected fields, removed keys, and known schema mismatches.

The classifications below are based on live reads and validation paths, not on
old plans or README claims. The audit mechanically parsed all 215 Python files
(37,110 physical lines), all five shell scripts (437 lines), both HTML dashboard
templates (784 lines), both TOML files, and all 79 current `config*.txt`
templates. Generated bytecode, local HDF5/checkpoints, output folders, and test
artifacts are not source configuration and were excluded.

## 1. Status legend

| Mark | Meaning | Should an authored config contain it? |
| --- | --- | --- |
| **R** | Required by both the suite and applicable runtime path. | Yes. |
| **R\*** | Runtime-required but missing from the suite's required-field check. | Yes; omission may pass preflight and fail later. |
| **C** | Conditionally required when a feature/mode is selected. | Only in that condition. |
| **O** | Optional and active; omission selects the shown native default. | Only when changing the default. |
| **M** | Compatibility/intent marker. Accepted or required by schema, but it does not select a live implementation. | Keep only when the schema requires it. |
| **D** | Derived/internal value injected by the runtime. | No. |
| **I** | Accepted but inert in the current live path. | No; remove it unless preserving an old file for history. |
| **L** | Legacy/removed and silently ignored by the named backend. | No. |
| **X** | Rejected/unsupported for the named backend or value. | No. |
| **G** | Schema gap: meaningful in code but cannot be safely authored through every launcher/validator path. | Usually no; see the exact note. |

“Required” is mode-specific. An inference file does not need training optimizer
controls, and an SDFFlow sampling file does not need VAE-training controls.

## 2. Flat-text config grammar

All five native loaders use the same basic parser behavior, and the suite parser
intentionally adds stricter diagnostics.

```text
key value
```

| Rule | Effective behavior |
| --- | --- |
| Key case | Keys are lowercased. `Training_epochs`, `training_epochs`, and `TRAINING_EPOCHS` become the same key. |
| String case | Non-numeric strings are also lowercased. This matters for case-sensitive paths on Linux. |
| Comments | A line beginning with `%` is ignored. Text after `#` is removed. Blank lines are ignored. |
| Separators | The first whitespace separates key from value. Tabs and spaces both work. |
| Lists | Commas or multiple space-separated tokens produce a list. Example: `gpu_ids 0,1`. |
| Booleans | Only case-insensitive `true` and `false` become booleans. |
| Numbers | A token containing `.` is parsed as float; otherwise integer parsing is attempted. Scientific notation such as `1e-4` remains a string, although many consumers later call `float(...)`. Prefer `0.0001`. |
| Quoting | Quotes are not syntax. They become literal characters in the value. Do not quote paths, even paths containing spaces. |
| Duplicate keys | Native loaders silently keep the last value. The suite reports a blocking duplicate-key error. Runtime override scripts that append duplicate lines therefore fail suite parsing. |
| `reserved` | A key named `reserved` is ignored. |
| Encoding | Use UTF-8 without BOM. The suite reports BOMs because native parsers misread the first key. |
| Invalid line | Native loaders generally skip a line lacking a value; the suite reports it as malformed. |

### 2.1 Path base and precedence

- The suite resolves the selected backend and launches it with that backend
  directory as the working directory.
- The original config file is passed unchanged. Relative paths inside it are
  therefore relative to `Geometry_generation`, `MeshGraphNets`,
  `MeshGraphNets - variational`, `Neural_Operator`, or `transolver`, not to the
  config file and not to the suite root.
- The centralized templates correctly use paths such as
  `../output/meshgraphnets/ex1/model1.pth` because the native process starts
  inside `MeshGraphNets`.
- For inference, architecture and normalization metadata are checkpoint-led:
  deterministic MGN, variational MGN, and Transolver overlay
  `checkpoint['model_config']`; Neural Operator rebuilds from its versioned
  `data_config`, `adapter_config`, and `model_config`. Config-file architecture
  values may be replaced after the checkpoint is loaded.
- Runtime-only choices such as inference dataset, output directory, rollout
  length, sample count, and sampling temperature remain config-controlled unless
  a backend explicitly persisted and overlays the same field.

## 3. Unified suite launcher

Canonical command:

```powershell
python CAE_ML_Suite_main.py --config configs\MeshGraphNets\ex1\config_train1.txt
```

### 3.1 Routing values

| `model` value | Backend | Entry point | Valid modes |
| --- | --- | --- | --- |
| `meshgraphnets` | deterministic MeshGraphNets | `MeshGraphNets/MeshGraphNets_main.py` | `train`, `inference` |
| `meshgraphnets-v` | variational MeshGraphNets | `MeshGraphNets - variational/MeshGraphNets_main.py` | `train`, `inference` |
| `point_deeponet` | Neural Operator | `Neural_Operator/main.py` | `train`, `inference` |

| `fno` | Neural Operator | `Neural_Operator/main.py` | `train`, `inference` |
| `gino` | Neural Operator | `Neural_Operator/main.py` | `train`, `inference` |

`MeshGraphNets-V` in a config is lowercased to `meshgraphnets-v` and works, but
underscore spellings are not aliases.

### 3.3 Local interpreter TOML

Copy `cae_suite.local.example.toml` to ignored `cae_suite.local.toml`.

```toml
[python]
default = "C:/Python311/python.exe"

[python.models]
meshgraphnets = "C:/envs/mgn/python.exe"
meshgraphnets-v = "C:/envs/mgn-var/python.exe"
neural_operator = "C:/envs/operators/python.exe"
transolver = "C:/envs/transolver/python.exe"
sdfflow = "C:/envs/sdfflow/python.exe"
```

Interpreter precedence is `--python`, exact model ID, method/spec ID,
`python.default`, then the suite process's `sys.executable`. The
`neural_operator` spec entry applies to all four operator model IDs unless an
exact `point_deeponet`, `deeponet`, `fno`, or `gino` entry is added.

## 12. Live mismatches and traps

These are current code facts, not hypothetical cautions.

1. **Centralized audit discovery is stale.** `--audit-configs` searches each
   backend for `config*.txt`; all 79 live files are now under root `configs/`,
   so the command reports zero.
2. **`num_workers` is runtime-required but under-specified.** MGN variants,
   Neural Operator, and Transolver training directly index
   `config['num_workers']`; suite required sets omit it. Always include it.
3. **Static rollout length is under-specified.** `infer_timesteps` can be
   inferred only from a temporal dataset with `T>1`; `T=1` rollout otherwise
   raises after preflight.
4. **SDFFlow guidance default differs.** Suite mode metadata says `cfg_scale
   2.0`; native `sample.py` actually uses `1.0` if absent because suite defaults
   are diagnostic and not injected.
5. **Variational weight decay is inert.** Suite accepts `weight_decay`, but
   variational training constructs Adam without it.
6. **Neural Operator markers/inert fields exist.** `coordinate_normalization`
   is a required marker; `train_query_chunk_size`,
   `gino_group_shared_geometry`, and `integration_weight_source` do not alter
   the current live path.
7. **Neural Operator hidden defaults cannot be overridden.** Code reads
   `gino_transform_type`, `train_eval_subset_size`, and `use_parallel_stats`,
   but the native unknown-key registry (and suite) omit them. Explicit lines are
   rejected, leaving implicit values `linear`, `128`, and `True`.
8. **Transolver visualization schema gap.** `test_batch_idx` is active natively
   but absent from the suite Transolver schema.
9. **Checkpoint metadata wins architecture conflicts.** Editing inference
   architecture lines does not reliably change the constructed model and may
   only create misleading files. Treat them as documentation/match checks.
10. **Cross-model Neural Operator keys are not shared hyperparameters.** They
    are recognized but inactive and only produce a notice.
11. **Direct-native and suite duplicate behavior differs.** Native parsers use
    last-wins; suite parsing blocks duplicates. This specifically affects the
    variational runtime override generator.
12. **Centralized B8 shell paths are stale.** Both runners still assume they
    live inside the native variational repository.
13. **Loss plot path logic is stale.** Plot helpers insert a GPU-ID directory
    that training log writers do not.
14. **Several removed docs/old templates are not part of this snapshot.** Do
    not reuse older counts such as 96 configs; the audited live count is 79.
15. **The elasticity configs are ahead of their implementation.** Their
    `split_strategy`, `loss_type`, and `relative_l2_epsilon` fields are unknown
    to suite schemas. Neural Operator native validation also rejects those three
    plus `use_parallel_stats`; Transolver ignores the three benchmark fields.
    Consequently the intended published split and relative-L2 objective are not
    active. The converted HDF5 is also absent in this snapshot.

## 14. Minimal authoring checklist

Before saving or launching a new config:

1. Choose the canonical `model`; do not infer routing from the folder name.
2. Choose a valid `mode` for that model.
3. Start from the closest current template in section 11, not an old backend
   README or removed config.
4. Include all R and R\* fields for the selected mode, especially
   `num_workers` and static-data `infer_timesteps`.
5. Keep paths backend-relative. From root templates, `../dataset` and
   `../output` are intentional because native cwd is the backend.
6. Remove keys belonging to another method/model.
7. Remove all L/I keys unless the table explicitly says an M marker is required.
8. For multiscale MGN, verify `len(mp_per_level) == 2*levels+1` and provide
   valid per-level coarsening/cluster data.
9. For inference, match the checkpoint's model family. Assume checkpoint
   architecture and normalization override the file.
10. Use decimal numeric spelling, unquoted paths, no duplicate keys, and UTF-8
    without BOM.
11. Run structural preflight on the individual file:

    ```powershell
    python CAE_ML_Suite_main.py --config configs\...\config.txt --check --skip-filesystem-check --skip-environment-check
    ```

12. Then run full `--check` with actual datasets/checkpoints/environment. A
    clean check probes only bounded metadata and still does not prove a full
    epoch or rollout.
